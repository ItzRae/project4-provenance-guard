# project4-provenance-guard

## Architecture Overview
When a user submits text through `POST /submit`, the server first validates the request and assigns a unique `content_id.` The text is then passed through two independent detection signals: an **LLM-based classifier using the Groq API** and a **stylometric heuristic analyzer**. Their scores are combined into a weighted confidence score, which is mapped to an attribution result and transparency label. The final decision is written to the audit log before being returned to the client.

If a creator believes the classification is incorrect, they can submit an appeal through `POST /appeal`. The system records the creator's reasoning, links the appeal to the original submission, updates the content status to under_review, and appends the appeal to the audit log.

## Detection Signals
---

### 1. LLM-Based Classification (Groq)

The first signal uses the Groq API to analyze the overall writing style and estimate the probability that a passage was generated by an AI language model.

It captures higher-level characteristics such as consistency of tone, semantic coherence, repetitive phrasing, and stylistic patterns that are difficult to express with simple rules.

#### Strengths

- Captures contextual writing patterns
- Adapts better to different writing styles than fixed heuristics
- Produces a confidence score with a short explanation

#### Limitations

- Because it relies on another language model, it may inherit that model's biases or incorrectly classify highly polished human writing as AI-generated.

### 2. Stylometric Heuristics

The second signal measures statistical properties of the text, including:

- sentence length variance
- type-token ratio (vocabulary diversity)
- punctuation density

AI-generated writing often exhibits more uniform structure than naturally written text, making these features useful as an independent signal

#### Strengths

- Fast to compute
- Fully explainable
- Independent from the LLM

#### Limitations

Stylometric features cannot understand meaning. Human writing can naturally appear very regular, while edited AI writing may intentionally increase variation.

---
## Confidence Scoring

The final confidence score combines both signals using a weighted average:

> confidence = (0.65 × LLM score) + (0.35 × Stylometric score)

The LLM signal receives a larger weight because it captures broader semantic and stylistic patterns, while the stylometric signal provides an independent structural perspective. Rather than forcing every submission into either "AI" or "Human," the system intentionally preserves uncertainty when the evidence is mixed. The resulting confidence score maps to three attribution categories:

| Confidence | Attribution  |
| ---------- | ------------ |
| 0.00–0.25  | Likely Human |
| 0.26–0.74  | Uncertain    |
| 0.75–1.00  | Likely AI    |

#### Example scores

| Example                        |  LLM | Stylometric | Final Confidence |
| ------------------------------ | ---: | ----------: | ---------------: |
| Clearly AI-style passage       | 0.81 |        0.62 |         **0.74** |
| Casual human-written paragraph | 0.40 |        0.47 |         **0.42** |

--- 

## Transparency labels

The system displays one of three labels depending on the confidence score:

### High-confidence AI
Displayed when the combined confidence score is 0.75–1.00.
> This content was classified as likely AI-generated with high confidence. This result is based on multiple automated detection signals and may be appealed by the creator.

### High-confidence Human
Displayed when the combined confidence score is 0.00–0.25
> This content appears to be human-written with high confidence. This assessment is based on multiple automated detection signals and is not a guarantee of authorship.

### Uncertain
Displayed when the combined confidence score is 0.26–0.74
> The system could not confidently determine whether this content was human-written or AI-generated. Readers should treat this label as additional context rather than a final judgment.

---

## Appeals Workflow

Creators can contest a classification using the `POST /appeal` endpoint.

Each appeal includes:

- the original content_id
- the creator's written reasoning

When an appeal is submitted, the system:

1. Locates the original submission.
2. Records the creator's reasoning.
3. Updates the submission status to under_review.
4. Stores the appeal alongside the original decision in the audit log.

This project does not automatically reclassify appealed content. Appeals are intended to be reviewed by a human.

## Rate Limiting

The submission endpoint is protected using `Flask-Limiter`

#### Limit

> 5 submissions per minute per IP address

This limit is intentionally conservative. It is high enough for normal use while discouraging large numbers of automated submissions that could overwhelm the detection service or abuse the API.

## Known Limitations

One type of content the system is likely to misclassify is carefully edited academic or professional writing. Formal human writing often shares characteristics with AI-generated text, such as consistent sentence structure and precise language, which can increase both the LLM and stylometric scores.

Similarly, heavily edited AI-generated text may intentionally introduce more varied sentence lengths and vocabulary, making it appear more human according to the stylometric signal.

This project should therefore be viewed as providing evidence for attribution rather than making definitive authorship claims.

## Spec Reflection

The project specification was helpful because it encouraged designing the overall architecture before writing implementation code. Planning the submission pipeline, confidence scoring, and audit logging first made it easier to build each milestone incrementally.

One place where my implementation diverged from the original plan was the first detection signal. During early development I used a simple placeholder heuristic to verify the API pipeline before replacing it with a Groq-based LLM classifier. This allowed me to test the surrounding infrastructure independently before integrating an external API.

## AI Usage

### Instance 1 – Flask API Skeleton

I prompted Claude to generate the initial Flask application skeleton based on my architecture diagram and API specification. The generated code included the `/submit, /appeal`, and `/log` endpoints, along with request validation and JSON responses.

After reviewing the output, I revised the request validation, updated the field names to match the project specification (such as using `creator_reasoning`), and modified the appeal workflow so it retrieves the original `creator_id `from the audit log instead of requiring it in the request.

### Instance 2 – Detection Pipeline

I used AI to help implement the stylometric detection function and the confidence-scoring pipeline

I adjusted the scoring thresholds, verified that the confidence calculation matched the weighting defined in my planning document (65% LLM, 35% stylometric), replaced the initial placeholder LLM classifier with a Groq API implementation, and corrected issues such as duplicate label-generation functions before integrating the pipeline into the submission endpoint.
